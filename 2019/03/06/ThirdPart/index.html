Data Type and Type Checking

Data type in programming languages

Static vs. dynamic data types

Static
即所有的变量类型在编译时（程序运行之前都是已知的），因此编译器可以推断出所有表达式的类型
Dyanamic
对于表达式类型是否正确的检查放在程序运行时（如Python）

Type checking

Static Checking and Dynamic Checking

  • 静态查错是指bug被自动的发现在程序运行前
  • 动态查错是指bug被自动的发现在编译时
  • No Checking需要自己检查并发现的错误，编译器并不能帮泥找出来

Static vs. Dynamic Checking

静态差错趋向于关于类型的错误，错误往往独立于变量的具体值
动态差错往往是由特定值引起的错误（如除0）

Mutability and Immutability

Changing a variable or its value

区别

  • 当你分配给变量时，泥将更改变量箭头指向（类似指针）指向的位置，可以将它指向不同的值
  • 当你分配可变值的内容时，泥将更改该值内的引用
    需要避免改变导致的内存泄露
    Java中用final修饰符表示不可变，需要注意的是：
  • final声明的class不能被继承
  • final声明的变量不能被改变
  • final声明的方法不能被子类覆盖
    String是一个不可改变的类型，在后面添加字符等于需要新建一个String
    StringBuilder 是一个可改变类型
使用可变类型的优势
  • 使用不可变字符会产生大量的临时拷贝
  • 使用可变类型可以减少拷贝
  • 从而能够获得更好的性能
  • 另一方面是方便共享，程序的两个部分可以通过共享一个可变数据结构通信（全局变量既视感）
使用可变类型的缺点
  • 传递可变值会导致该值在函数内改变影响结果
  • 返回一个可改变的值

解决方法
Defensive Copying
可以拷贝一遍可变类型然后再返回拷贝后的结果

Snapshot diagram as a code-level,run time, and moment view


快照图为我们提供了一种可视化更改变量和更改之之间区别的方法：当你分配变量或字段时，将改变箭头指向的位置。你可以将它指向不同的值。当你改变值的内容时如数组或者LIST泥改变的是这个值的内在引用。

  • 对于一个对象类型的值在快照图中用一个圈表示这个对象，当我们想要显示更多细节时，在其中写入字段的名字箭头只想他们对应的值
  • 不可变对象在快照图中用双边框表示
  • 不可变对象的每次更改需要新建一个
  • 在快照图中一个不可改变的引用(final)用双箭头表示

Complex data types: Arrays and Collections

  • iterator中next()函数时可变类型
  • 变异破坏了迭代器
  • iter.remove()可以安全的删除数据，注意iter.remove只能删除iter.next()返回的数据

尽可能使用不可变对象和不可变引用
注意不可变对象和不可变对象引用的区别

Specification: Programming for communication

Documenting in programming

  • 类的层次结构以及已经实现的接口
  • 直接子类，以及实现的接口
  • 类的概述
  • 构造函数的摘要
  • 构造函数摘要
  • 列出我们可以调用的所有函数
  • 对每个方法和构造函数进行详细描述
    • 返回类型，函数名称，传递的参数变量以及异常
      Documenting Assumptions
  • 写下声明的变量类型 功能
  • 声明一个变量是final类型

    Specification and Contract(of a method)

    Why
    Reality:
  • 程序中很多棘手的问题都是由于误解两端代码直接的接口造成的
  • 尽管每个程序员都考虑到了规范，但并非每个程序员都能够将其记录下来
  • 因此，团队中不同的程序员会考虑不同的规范
  • 进而当程序运行失败时，很难确定错误的位置
    Advantages
  • 代码中的精确说明帮助你将每一项责任归因于代码的每一部分
  • 代码说明可以帮助客户端的用户使用(他们只需要使用而不需要读写代码)
  • 代码的可以根据说明随意的更改具体实现方式而不需要告诉客户
  • 说明可以使coding更迅速

    Behavioral equivalence

    对于函数的说明不能涉及局部变量

    Specification structure: pre-condition and post-condition

  • precondition,由关键字requires表示
  • postcondition,由关键字effec他表示
  • 异常行为:如果违背了前提条件会发生什么

Java的静态类型声明是前置条件和后置条件的一部分，他是由编译器自动检查和强制执行的部分。
(precondition)编写程序时用@param描述变量
(postcondition)用@return和@throws描述结果。

  • 在spec中不应该设计局部变量和类中的私有部分
  • 除非在后置条件里声明过，否则方法内部不应该改变输入参数
  • 应该尽量不涉及mutating的spec，否则容易引发bugs
  • 尽量避免使用mutable对象

Testing and verifying specificatons

Javadoc
记录每个变量，返回值，异常(已检查的未检查的)，这个函数的作用包括目的，副作用，任何线程安全问题，以及性能问题
不对具体实现细节描述

Designing specifications

Classifying specifications

Comparing specifications

  • 规约的确定性
  • 规约的陈述性

  • 规约的强度

    Stronger vs. weaker specs

    如果前置条件更弱或者后置条件更强那么就认为这个规约更强
    可以用较强的规约代替较弱的规约
    如果两个规约之间没有确定的强弱关系就认为他们incomparable

    Deterministic vs. underdetermined specs

    确定的规约：即给定一个满足precondition的输入，其输出是唯一的、明确的
    欠定的规约：同一个满足条件的输入，其输出可能有多种
    非确定的规约： 对于同一个输入，多次执行时可能的到的输出不同
    欠定的规约通常有确定的实现

    Declarative vs. operational specs

    操作式规约：例如伪代码
    声明式规约: 没有内部实现的描述，只有初始和最终状态
    声明式规约更有价值
    内部实现的细节不在规约里面呈现，放在代码实现体内部注释里呈现

  • 操作性规范给出了一系列该方法执行的步骤

  • 声明性规范不提供中间步骤的详细信息，他只关注最终的结果属性以及与出事状态的关系
