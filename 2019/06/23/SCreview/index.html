2.2 Process, Systems, and Tools of Software Construction

  • 代码评审分为：结对编程、走查、正是评审回忆、自动化评审
  • java 使用jar文件的命令 创建 jar -tf example.jar 使用 java -cp example.jar com.examples.Main
  • 软件构造得一般过程：设计、编码、重构、debug、测试、构建、发布
  • 狭义得软件构造过程: 合法性、编译、链接、测试、打包、安装、部署

3.1 Data Type and Type Checking

  • 基本数据类型、对象数据类型
  • 静态类型语言在编译阶段进行检查，动态类型语言在运行阶段进行检查
  • 改变一个变量: 将该变量指向另一个值得存储空间
  • 改变一个变量得值: 将该变量指向得值得存储空间种写入一个新的值
  • 拷贝式编程
  • 引用是不可改变的但是指向的值是可以改变的，可变的引用可以指向不可变的值.
  • snapshot 图

3.2 Designing Specification

  • 更弱的前置条件和更强的后置条件都会使 spec变强
  • 确定的规约(给定输入输出确定) 欠定的规约（给定输入可以有多个输出） 非确定的规约（给定输入多次执行输出可能不同）
  • 操作式规约 声明式规约
  • 是否适用前置条件取决于（1) check的代价 （2） 方法的使用范围

3.3 Abstract Data Type (ADT)

  • 抽象：使用更简单，更高级别的想法省略或隐藏低级细节
  • 模块化：将系统划分为组件或模块，每个组件或模块可以与系统的其余部分分开设计，实现，测试，推理和重用
  • 封装：围绕模块（硬壳或胶囊）构建墙壁，以便模块负责其自身的内部行为，并且系统其他部分中的错误不会损害其完整性
  • 信息隐藏：从系统的其余部分隐藏模块实现的详细信息，以便稍后可以更改这些详细信息，而无需更改系统的其余部分
  • 关注点分离：使功能（或“关注”）成为单个模块的责任，而不是将其分布在多个模块中
  • ADT 由其操作定义 与其内部如何实现无关
  • 构造器 创建给类型的新对象
  • 生产器 以旧的对象为依据/参数创建一个新的对象
  • 观察器 传入抽象类型对象返回不同类型对象 eg. list.size()
  • 变值器 改变对象属性的方法(通常返回void 可以返回非空那个类型)
  • AF 说明每个变量怎么转化到它的抽象意义 RI 说明了一个合法的表示值（如没有重复字符的字符串)
  • 测试creators, producers, and mutators：调用observers来观察这些 operations的结果是否满足spec; 测试observers：调用creators, producers, and mutators等方法产生或改变对象，来看结果是否正确

3.4 Object-Oriented Programming (OOP)

  • 一个类可以实现多个接口

  • 使用静态工厂使得用户不用知道接口的某个具体实现类的名字

  • 抽象方法 定义了传入参数但是没有实现

  • 抽象类 至少包含一个抽象方法的类
  • 重载 == 静态多态 （在编译时决定具体执行哪个方法)
  • 重载一定要改变参数列表
  • 子类型的规约不能弱化超类型的规约
  • 动态分配 推迟绑定 运行阶段决定使用哪种子方法
  • OOP的历史在这里（可能要看？

3.5 Equality in ADT and OOP

  • AF定义等价性（映射到同样的结果则等价）
  • 站在外部观察者角度：对两个对象调用任何相同的操作，都会得到相同的结果，则认为这两个对象是等价的
  • 不要在超类种用instanceof检查子类
  • equals满足自反对称传递
  • 观察等价性：在不改变状态的情况下，两个mutable对象看起来是否一致
  • 行为等价性：调用对象的任何方法都表现出一致的结果
  • 可变类型在放入HashSet,HashMap等结构的同时即确定了hash值不再改变即如果改变了可变类型就在set/map里找不到了
  • 不可变类型必须重写equals和hashcode，可变类型不应该重写这两个方法
  • x.clone() != x

4.1 Construction for Understandability

  • 命名长度、命名独特性比例、代码长度、注释的密度(MCOMM %)、代码布局、避免多层嵌套、等宽字体
  • 标题注释、介绍性注释、块注释、行注释
  • 注释说明做什么不说明怎么做、注释不应该展示代码中可以被轻易推测的信息
  • 复用的粒度应等价于 发布的粒度
  • 不允许在包依赖 图中出现任何圈/回路
  • 包之间的依赖关系只能指向稳定的方向,稳定的包 较难发生改变
  • 在稳定性与抽象度之间建立关联,一个完全稳定的包中只应包含抽象类,一个包是稳定的，那么它就 应该尽可能抽象
  • CRP原则: 一个包中所有类应该是共同重用的。如果重用了包中的一个类，那么就要重用包中的所有类

5.1 Metrics, Morphology and External Observations of Reusability

  • 框架：一组具体类、抽象类、及其之间的连接关系
  • 开发者根据 framework的规约，填充自己的代码进去，形成完整系统（比如gui中实现点击屏幕某一位置的功能）
  • 控制反转 由第三方容器来控制对象之间的依赖关系
  • LSP 里氏替换原则

5.2 Construction for Reuse

  • 泛型不协变—类型擦除
  • 有通配符忽视上面一条
  • 如果子类只需要复用父类中的一小部分方法可以不需 要使用继承，而是通过委派机制来实现
  • 接口之间通过extends实现 行为的扩展（接口组合）
  • 白盒设计模式 使用子类化和重写方法进行扩展 通常使用模板设计模式
  • 黑盒设计模式 使用插件接口扩展 通常使用 strategy observer设计模式
  • iterator 本质是复制了一份引用所以可以remove

6.1 Metrics and Construction Principles for Maintainability

  • 圈复杂度、代码行数、继承的曾次数、类之间的耦合度、单元测试的覆盖率用来衡量可维护性的方法
  • 模块化的五个要求：可分解性、可组合性、可理解性、可持续性、出现异常之后的保护
  • 模块化设计的五个规则：直接映射、尽可能少的接口、尽可能小的接口、显式接口、信息隐藏.
  • 单一责任原则（让修改某个类的方法尽量少）
  • 开闭原则 （对扩展性开放，对修改封闭）
  • 客户端不需要的接口就不用给它 使用private保护起来

6.3 Maintainability-Oriented Construction Techniques

  • Table-Driven 将代码中复杂的if-else和switch-case语句从 代码中分离出来，通过“查表”的方式完成，从而提高可维护性

7.1 Robustness and Correctness

  • 对外的接口，倾向于健壮；对内的实现，倾向于正确
  • MTBF 平均失效间隔时间
  • KLOC 每千行代码中遗留的bug数量

7.2 Error and Exception Handling

  • errors and runtime exceptions are unchecked exceptions
  • throws使用在函数上，throw使用在函数内
  • 根据MIT规则 对特殊的返回结果使用checked异常 对bug、未知的错误使用unexpected异常
  • 程序员必须在方法的spec中明确写清本方法会抛出的所有checked exception，以便于调用该方法的client加以处理
  • 对于异常 尽量在函数内部处理

  • finally 不管怎么样都会被运行

  • TWR 退出try block的时候自动close 不用写finally

7.3 Assertions and Defensive Programming

  • java -da test //取消断言
  • 程序之外的事情不要乱做断言
  • 防止不合法输入、断言、抛出异常、解决error –> 防御式编程
  • 操作间计数？ proxy设计模式？ 对外部函数的调用使用异常处理、对内部函数的调用使用断言

7.4 Debugging

  • 输出调试 经常用
  • 分治查错 sb玩意，是在找不出来可以用一用，但是bug是很多个连锁就gg
  • 切片差错、观察代码每一行对出错的部分是否有贡献
  • 利用版本控制寻找在哪个commit后出现bug 然后对比代码改变 // git bisect
  • 符号化调试
  • Debugger 不会用.jpg

7.5 Testing and Test-First Programming

  • 白盒测试：对程序内部代码 结构的测试
  • 黑盒测试：对程序外部表现出来的行 为的测试
  • 针对软件进行单元测试
  • 可以使用等价类的思想
  • 注意边界情况（要是经常在这里犯错不妨对所有边界特殊处理
  • 白盒测试考虑内部实现细节更容易debug但是时间消耗很大

8.2 Memory Performance and Garbage Collection

  • 所有局部的基本数据类型都在栈上创建
  • 多线程 之间传递数据，是通过复制而非引用
  • 所有对象都在堆上创建
  • 堆上创建的对象可被所有线程共享引用
  • 可访问对象，就可以访问对象内的成员 变量
  • 如果两个线程调用同一个对象上的 某个方法，它们分别保留该方法的 局部变量的拷贝
  • Method Area: 用于存储被VM加 载的类信息、常量、静态变量等
  • gc四种策略 引用计数（拓扑） 标记-清楚（bfs) 标记整理(空余出来的放到一起) 复制（把要用的复制出来、剩下的全删了)
  • 困难的参数设置
    -XX: MetaspaceSize perm 初始size
    -XX: MaxMetaspaceSize perm 初始size
    -Xms: 初始整体内存大小
    -Xmx: 整体最大大小
    -XX: NewSize=[g|m|k] 表示yonng的大小
    -XX MaNewSize=[g|m|k] 表示yong的最大大小
    -XX:NewRatio= the radio between and old generation is 1 : n
    -XX:SurvivorRatio= the radio between eden and survivor space n : 1
