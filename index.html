Data Type and Type Checking

Data type in programming languages

Static vs. dynamic data types

Static
即所有的变量类型在编译时（程序运行之前都是已知的），因此编译器可以推断出所有表达式的类型
Dyanamic
对于表达式类型是否正确的检查放在程序运行时（如Python）

Type checking

Static Checking and Dynamic Checking

  • 静态查错是指bug被自动的发现在程序运行前
  • 动态查错是指bug被自动的发现在编译时
  • No Checking需要自己检查并发现的错误，编译器并不能帮泥找出来

Static vs. Dynamic Checking

静态差错趋向于关于类型的错误，错误往往独立于变量的具体值
动态差错往往是由特定值引起的错误（如除0）

Mutability and Immutability

Changing a variable or its value

区别

  • 当你分配给变量时，泥将更改变量箭头指向（类似指针）指向的位置，可以将它指向不同的值
  • 当你分配可变值的内容时，泥将更改该值内的引用
    需要避免改变导致的内存泄露
    Java中用final修饰符表示不可变，需要注意的是：
  • final声明的class不能被继承
  • final声明的变量不能被改变
  • final声明的方法不能被子类覆盖
    String是一个不可改变的类型，在后面添加字符等于需要新建一个String
    StringBuilder 是一个可改变类型
使用可变类型的优势
  • 使用不可变字符会产生大量的临时拷贝
  • 使用可变类型可以减少拷贝
  • 从而能够获得更好的性能
  • 另一方面是方便共享，程序的两个部分可以通过共享一个可变数据结构通信（全局变量既视感）
使用可变类型的缺点

Software Development Lifecycle(SDLC)

  1. planning
  2. analysis
  3. design
  4. implementation
  5. testing & integration
  6. maintenance

Traditional Software process models

两种基本类型

  • 线性过程
  • 迭代过程

存在的模型

  • 瀑布过程
  • 增量过程
  • V字模型
  • 原型过程
  • 螺旋模型

选择合适的过程模型的依据

  • 适应变化的能力
  • 开发效率/管理复杂度
  • 开发出的软件的质量

瀑布模型

通过概念，启动，分析，设计，构建，测试，实施和维护阶段，进展被视为稳步向下（如瀑布）。

瀑布的过程
  • 线性推进
  • 阶段划分清楚
  • 整体推进
  • 无迭代
  • 管理简单
  • 无法适应需求增加/变化

增量模型

将软件的开发分成多个阶段 产品经过设计，实施和逐步测试（每次添加一点），直到产品完成
用户可以增加新的东西但不能更改原有的东西

增量过程
  • 线性推进
  • 增量式（多个瀑布的串行）
  • 无迭代
  • 比较容易适应需求的增加

原型过程

迭代：开发出来之后由用户试用/评审，发现问题反馈给 开发者，开发者修改原有的实现，继续交给用户评审。
循环往复这个过程，直到用户满意为止。 时间代价高，但开发质量也高

螺旋模型

循环往复更新某些功能

过程（复杂）
  • 多轮迭代基本遵循瀑布模式
  • 每轮迭代有明确的目 标，遵循“原型”过 程，进行严格的风险 分析，方可进入下一 轮迭代

V字模型

从完成开发的最小单元开始进行测试

  • 在编码阶段之后，工艺步骤向上弯曲，而不是以线性方式向下移动，以形成典型的V形
  • 展示开发生命周期的每个阶段与其相关测试阶段之间的关系。
  • 水平轴和垂直轴分别表示时间或项目完整性（从左到右）和抽象级别（最粗糙的抽象最粗）。

Agile development

通过快速迭代和小规模 的持续改进，以快速适应变化
Agile = 增量 + 迭代 每次迭代处理一个增量

Waterfall VS Agile

  • 瀑布模型 只有依次交付 顾客只能提一次要求
  • 敏捷模型多次迭代 多次交付 用户提多次要求 每次迭代进行多次更新修改

Software Configuration Management (SCM) and Version Control System (VCS)

SCM: 追踪和控制软件的变化
SCI：软件中发生变化的基本单元（如：文件）
基线(baseline)： 软件持续变化过程中的“稳定时刻”（例如：对外发布的版本）
CMDB：配置管理数据库
存储软件的各配置项随时间发生变化的信息+基线

软件构造过程中的多维度视图

多维度软件视图

按阶段划分： build and run-time vies (构造时/运行时视图)
按动态性划分： moment and eriod views （时刻/阶段视图）
按构造对象划分： code and component views （代码/构件视图）

需要关注的是每个视图的元素、关系和模型

build-time

代码级视图：源代码 源代码如何通过基本程序块（如函数，类，方法，接口等）进行逻辑组织，以及它们之间依赖关系代码的逻辑组织
组件级视图：体系结构–源代码如何按文件、目录、包、库以及它们之间的一拉关系进行物理组织代码的物理组织
时刻视图：源代码和组件在特定时间内的样子特定时刻的软件形态
期间视图：它们如何随着时间的推移而发展/变化。软件形态随时间的变化

Build-time,moment and code-level view的结合

即从源代码的角度，从其基本的程序模块（包括函数，类，方法，等等）以及他们之间的联系，看源代码的逻辑组织。
具有三个内在联系的形式：词汇层面，语法层面，语义层面。
词汇层面： 使用半结构化的代码：近乎自然语言的风格+遵循特定的编程语法。
语法层面：常用抽象语法树（AST），来表示一个半结构化的源代码。
语义层面：考虑源代码要实现的目标，将源代码与现实世界联系起来，使用图形化或形式化的方法，来表达”需求“与”设计“思想，再将其转化为code。主要为面向对象的分析与面向需求的设计。
用于表达“需求”和设计思想，再转化为code

build-time,period and code-level view

随着时间观察变化
例如code churn(代码变化)

build-time,moment and component view

从代码的物理结构（源代码、文件、包和库等)看代码的物理组织

  • 源代码再物理结构上一文件的形式组织，更上层是以目录形式组织
  • 文件再逻辑上被封装为Package，compones和子系统
  • 可服用的模块以库的形式存储
    在执行build-time的时候需要执行静态链接，在执行时无需提供库文件，动态链接时在run-time时执行的

库

库存储在自己的磁盘文件中，持有一组可以被其他程序调用的代码函数
开发人员并不总是构建单个的可执行程序文件，而是将自定义开发的软件和预构建的库放到单个程序中去

库的来源
  • 操作系统提供
  • 编程语言提供
  • 第三方公司
  • 自己编写

Maven repositor一个JAVA的开源项目

build-time,period and component-level view

这一方面主要关注的是软件实体随时间变化的趋势，主要体现在版本和SCI上

Runtime Views

运行时即程序被载入目标机器开始运行的时候
代码层面：即逻辑实体如何在内存中呈现
构件层面：即物理实体在物理硬件环境重如何呈现
时刻层面：逻辑/物理实体在内存/环境中特定时刻的形态变化
阶段层面：逻辑/物理实体在内存/环境中特定随时间的形态变化

从run-time角度看软件的一些高级概念

  • 可执行程序： CPU执行的机器可读指令序列以及相关的数据值
    • 原生机器码
  • 库： 可以由不同程序使用的常用对象的集合
  • 配置和数据文件：提供可以从程序可以加载的有用数据和配置信息，是不可执行文件
  • 分布式程序： 这种类型的软件可由多个可执行程序组成，这些程序可以通过网络相互通信，或者只是在同一台机器上运行多个进程

ArrayList 与 LinkedLIst

ArrayList 它长于随机访问元素，但是在List中间插入和删除元素比较慢（感觉看上去像是vector）
LinkedList 它通过代价较低的在List中间进行的插入和删除操作，提供了我优化的顺序访问，LinkedList在随机访问方面相对比较慢。

ListIterator

ListIterator是一个更强大的Iterator的子类型，它只能用于各种List类的访问。
其特点是可以实现双向移动，同时可以使用set()的方法替换它访问过的最后一个元素

Set
